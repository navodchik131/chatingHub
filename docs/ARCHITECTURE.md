# Архитектура Chating Hub (SaaS)

Документ описывает целевую мультитенантную систему: личный кабинет, подписка, кредиты, интеграции Telegram / Fanvue, деплой в Docker.

## 1. Границы системы

| Компонент | Назначение |
|-----------|------------|
| **API (FastAPI)** | Auth, бизнес-логика, вебхуки внешних платформ, WebSocket уведомлений |
| **PostgreSQL** | Пользователи, подписки, кредиты, диалоги, зашифрованные секреты интеграций |
| **Фронтенд (React SPA)** | Личный кабинет, чат, настройки интеграций, оплата (редирект в Stripe Checkout) |
| **Stripe** | Оформление подписки, вебхук статуса, при необходимости — пополнение кредитов |
| **Telegram Bot API** | Входящие/исходящие сообщения Direct messages канала (режим **webhook** в проде) |
| **Fanvue API** | Входящие вебхуки + исходящие сообщения от имени создателя |

Локальная разработка без Docker может использовать SQLite (как раньше); **продакшен-цель — PostgreSQL**.

## 2. Мультитенантность

- Каждый **пользователь** (`users`) изолирован: все `conversations` и `messages` имеют `user_id`.
- **Интеграции** хранятся отдельно:
  - `telegram_connections` — токен бота (шифрование), секрет пути вебхука, флаги активности.
  - `fanvue_connections` — OAuth access token (шифрование), UUID создателя (`creator_uuid` для маршрутизации вебхуков), секрет подписи вебхука Fanvue (шифрование).
- Публичные HTTP-вебхуки **не требуют JWT**; идентификация арендатора:
  - Telegram: непредсказуемый сегмент URL `/api/webhooks/telegram/{webhook_secret}`.
  - Fanvue: тот же паттерн + проверка подписи секретом из записи пользователя.

## 3. Аутентификация

- Регистрация / вход по email + пароль (bcrypt).
- Ответ: **JWT** (Bearer), срок жизни задаётся конфигом (`JWT_EXPIRE_MINUTES`).
- Защищённые маршруты: `Authorization: Bearer <token>`.
- WebSocket: тот же токен в query `?token=` (упрощённый MVP; дальше — короткоживущий WS-токен).

## 4. Подписка и кредиты

### Подписка (Stripe)

- Модель `subscriptions`: `stripe_customer_id`, `stripe_subscription_id`, `status`, `plan_tier`, `current_period_end`.
- `POST /api/billing/checkout` — создаёт **Checkout Session** (режим subscription), `success_url` / `cancel_url` с фронта.
- `POST /api/billing/webhook` — проверка подписи Stripe, обновление статуса подписки.

### Кредиты

- `credit_accounts.balance` — целочисленные «кредиты» (условные единицы).
- `usage_events` — журнал списаний/начислений (аудит).
- Списание (MVP): за перевод входящего и за перевод исходящего ответа (конфиг `CREDIT_COST_INBOUND_TRANSLATION`, `CREDIT_COST_OUTBOUND_TRANSLATION`).
- **Политика доступа (MVP):** активная подписка (`active` / `trialing`) **или** положительный баланс кредитов — иначе `402` на операциях, требующих перевод/отправку.

Планы расширения: пакеты кредитов через Stripe Payment Links / Checkout `mode=payment`, роли, команды (организации).

## 5. Telegram (продакшен)

- Для SaaS используется **webhook** на каждого пользователя: при сохранении токена вызывается `setWebhook` на `PUBLIC_APP_URL/api/webhooks/telegram/{secret}`.
- Требуется **HTTPS** и корректный `PUBLIC_APP_URL`.
- Старый режим глобального `BOT_TOKEN` + long polling в `.env` оставлен как **legacy** только для локальной отладки (`LEGACY_BOT_TOKEN`); в Docker по умолчанию не используется.

## 6. Fanvue

- У пользователя одна запись `fanvue_connections` (расширение: несколько приложений).
- Исходящие сообщения используют **токен из БД**, не глобальный env.
- Вебхук Fanvue настраивается в кабинете разработчика на URL с секретом пользователя.

## 7. Realtime

- `RealtimeHub` хранит WebSocket-клиенты по `user_id`.
- События `new_message` уходят только в сокеты владельца диалога.

## 8. Docker

- `docker-compose.yml`: сервисы `db` (PostgreSQL) и `api` (образ с фронтом внутри статики FastAPI).
- Переменные: см. `deploy/env.example` (корень репозитория) / секцию в README.

## 9. Безопасность (чеклист)

- `JWT_SECRET`, `FERNET_KEY` — длинные случайные значения, не коммитить.
- Секреты платформ только в зашифрованном виде (Fernet + ключ из env).
- Stripe webhook — проверка подписи.
- Fanvue webhook — проверка подписи ключом пользователя.
- Rate limiting и CAPTCHA на `/auth/*` — рекомендуется на краю (nginx / Cloudflare) в проде.

## 10. Эволюция

1. Вынести перевод в очередь (Redis + worker) при росте нагрузки.
2. Отдельный сервис биллинга / отчёты по `usage_events`.
3. Несколько ботов Telegram на пользователя; несколько каналов.
4. Организации и приглашения (RBAC).
